## Возможности

### Для рекрутера
- Выбрать клиента из списка и рассчитать бонус по его формуле
- Результат отображается сразу для двух ставок рекрутера — 20% и 30%
- Числа выводятся в читаемом формате: `150 000.00` вместо `150000`

### Для администратора
**Управление клиентами:**
- Добавить нового клиента (только имя, формула привязывается отдельно)
- Привязать или обновить формулу расчёта для клиента
- Удалить клиента (с подтверждением)
- Просмотреть список всех клиентов

**Управление администраторами:**
- Добавить нового администратора по Telegram ID
- Удалить администратора
- Просмотреть список всех администраторов

## Архитектура

Проект разделён на слои с чёткими зонами ответственности:

```
Telegram API
     ↓
startbot — получает обновления, передаёт в Bot
     ↓
telegram/handler.go — маршрутизация, бизнес-логика диалога
     ↓            ↓
parsers        database
(парсинг       (запросы
 ввода)         к БД)
     ↓
calculator — чистая математика формул
     ↓
utils — форматирование вывода
```

**Ключевые принципы:**
- `database` — только SQL-запросы, возвращает `error`, не формирует текст для пользователя
- `parsers` — только парсинг строк, не знает о БД и Telegram
- `calculator` — только математика, не знает ни о чём кроме чисел
- `handler.go` — единственное место где слои соединяются

### State Machine

Навигация реализована через стек меню (`StackMenu []Menu`). Каждый переход вглубь кладёт меню в стек, кнопка "Назад" достаёт предыдущее:

```
[MainMenu] → PushMenu(AdminMenu) → [MainMenu, AdminMenu]
                                          ↓ PopMenu()
                                   [MainMenu] → SendMainMenu
```

Текущее действие пользователя хранится в поле `Action` — это позволяет `HandleMessage` понимать контекст ввода:

```
ActionWaitClientName  — пользователь должен ввести имя клиента
ActionWaitSalary      — пользователь должен ввести зарплату кандидата
ActionWaitAdminID     — пользователь должен ввести ID нового администратора
```

### Динамические callback-данные

Для кнопок с динамическим содержимым (список клиентов) используется формат `"prefix:context:id"`:

```
"client:bonus:42"   — выбран клиент ID=42 для расчёта бонуса
"client:update:42"  — выбран клиент ID=42 для обновления формулы
"client:delete:42"  — выбран клиент ID=42 для удаления
"confirm:delete:42" — подтверждение удаления клиента ID=42
```

## База данных

### Схема

```
                    ┌───────────────────┐
                    │      admins       │
                    ├───────────────────┤
                    │ id (BIGINT) PK    │  ← Telegram ID
                    │ name (TEXT)       │
                    └───────────────────┘

          ┌─────────────────────────────────────┐
          │               clients               │
          ├─────────────────────────────────────┤
          │ id (SERIAL) PK                      │
          │ name (TEXT)                         │
          │ formula_type (ENUM) ←── NULL если   │
          └──────────────┬──────────────────────┘
                         │  ON DELETE CASCADE
          ┌──────────────┼──────────────┐
          │              │              │
          ▼              ▼              ▼
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│ standard_formula │  │ salary_formulas  │  │  free_formulas   │
├──────────────────┤  ├──────────────────┤  ├──────────────────┤
│ client_id PK,FK  │  │ client_id PK,FK  │  │ client_id PK,FK  │
│ percent NUMERIC  │  │ coeff.. NUMERIC  │  │ formula TEXT     │
└──────────────────┘  └──────────────────┘  └──────────────────┘
  ЗП × 12 × %           ЗП × коэф.            произвольная
```



```sql
-- Администраторы бота (Telegram ID как первичный ключ)
admins (
    id   BIGINT PRIMARY KEY,  -- Telegram ID пользователя
    name TEXT
)

-- Клиенты агентства
clients (
    id           SERIAL PRIMARY KEY,
    name         TEXT,
    formula_type formula_kind  -- ENUM: standard | salary | free | NULL
)

-- Формула: процент от годового дохода (ЗП × 12 × %)
standard_formulas (
    client_id INTEGER PRIMARY KEY REFERENCES clients(id) ON DELETE CASCADE,
    percent   NUMERIC NOT NULL
)

-- Формула: коэффициент от оклада кандидата (ЗП × коэффициент)
salary_formulas (
    client_id   INTEGER PRIMARY KEY REFERENCES clients(id) ON DELETE CASCADE,
    coefficient NUMERIC NOT NULL
)

-- Формула: произвольная (для будущей интеграции с ИИ)
free_formulas (
    client_id INTEGER PRIMARY KEY REFERENCES clients(id) ON DELETE CASCADE,
    formula   TEXT NOT NULL
)
```

### Особенности

- `formula_type` может быть `NULL` — клиент добавляется без формулы, формула привязывается отдельно
- `ON DELETE CASCADE` — при удалении клиента его формула удаляется автоматически
- `ON CONFLICT DO UPDATE` — добавление и обновление формулы через один UPSERT-запрос
- БД и все таблицы создаются автоматически при первом запуске через `InitSchema()`

### Формулы расчёта бонуса

| Тип | Формула | Описание |
|---|---|---|
| `standard` | `ЗП × 12 × %` | Процент от годового гросс дохода |
| `salary` | `ЗП × коэффициент` | Кратно окладу кандидата (1.0, 0.5, 2.0...) |
| `free` | произвольная | Индивидуальный расчёт, планируется интеграция с ИИ |

Результат всегда показывается для двух ставок рекрутера одновременно: **20%** и **30%** от чека.

## Запуск

### Локально

**Требования:** Go 1.23+, PostgreSQL 16+

```bash
# 1. Клонировать репозиторий
git clone <repo-url>
cd workwavebot

# 2. Заполнить переменные окружения
cp .env.example .env
# отредактировать .env

# 3. Запустить
go run ./cmd/workwavebot/
```

База данных и все таблицы создадутся автоматически при первом запуске.

### Docker Compose

```bash
# Заполнить .env, затем:
docker-compose up --build
```

Compose поднимает два контейнера: `postgres_db` и `telegram_bot`. Данные PostgreSQL сохраняются в volume `postgres_data`.

## Переменные окружения

Файл `.env` в корне проекта:

```env
BOT_TOKEN=your_telegram_bot_token

DB_USER=your_db_user
DB_PASSWORD=your_db_password
DB_HOST=localhost        # для Docker: имя сервиса из compose (db)
```

> `.env` не должен попадать в систему контроля версий. Добавьте его в `.gitignore`.

### Первый администратор

Первого администратора нужно добавить вручную напрямую в БД — это сделано намеренно, чтобы исключить несанкционированный доступ:

```sql
INSERT INTO admins (id, name) VALUES (ваш_telegram_id, 'Имя');
```

Telegram ID можно узнать через бота [@userinfobot](https://t.me/userinfobot).

## Навигация в боте

```
/start
└── Главное меню
    ├── Меню Рекрутера
    │   ├── Рассчитать бонус
    │   │   └── [список клиентов] → ввод ЗП → результат
    │   └── Полезности
    └── Меню Админа  [только для администраторов]
        ├── Управление клиентами
        │   ├── Показать всех клиентов
        │   ├── Добавить клиента       → ввод имени
        │   ├── Обновить формулу       → выбор клиента → выбор типа → ввод значения
        │   └── Удалить клиента        → выбор клиента → подтверждение
        └── Управление администраторами
            ├── Список всех администраторов
            ├── Добавить администратора  → ввод "ID Имя"
            └── Удалить администратора   → ввод ID
```
